#### open() 与文件模式 (r / w / a)

> **痛点**

程序运行结束,内存中的所有变量都会消失。
想要永久保存数据,必须写入硬盘上的文件。

> **类比**

文件就像一本笔记本。打开之前要决定做什么:
只读(r)、重新写一整本(w)、还是接着末尾写(a)。

> **解释**

`open(路径, 模式)` 返回一个文件对象,
是 Python 与硬盘文件之间的桥梁。三种基本模式:
- `r` 只读,文件不存在报错
- `w` 覆盖写,文件不存在则新建
- `a` 追加写,文件不存在则新建

```python
# 示例:open() 写入和读取
# 'w' 写入模式:覆盖原文件内容
f = open('demo.txt', 'w', encoding='utf-8')
f.write('第一行\n')
f.write('第二行\n')
f.close()

# 再次以 'r' 模式打开并读取
f = open('demo.txt', 'r', encoding='utf-8')
print(f.read())
f.close()

# 'a' 追加模式:在末尾添加,不会覆盖已有内容
f = open('demo.txt', 'a', encoding='utf-8')
f.write('第三行(追加)\n')
f.close()

# 逐行解剖
# ① open(路径, 模式, encoding) 打开文件
# ② f.write(s) 写入字符串,不会自动加 \n
# ③ f.close() 关闭文件,释放资源
```

**常见错误**

> 1. **错误现象**:忘写 encoding,中文乱码
>    **原因:**Windows 默认 gbk,macOS 默认 utf-8
> 2. **错误现象**:用 'w' 模式打开已有文件,内容被清空
>    **原因:**'w' 每次打开都会清空原内容

#### 三种读法: read / readline / readlines

> **痛点**

写进文件的数据还要读回来,
但文件大小不同,有时要一次性读,有时要逐行处理。

> **类比**

读一本书:read 等于一口气读完,
readline 等于一行行慢慢读,
readlines 等于先抄下所有行的目录再逐条读。

> **解释**

Python 提供三种读法:
- `read()`:读取全部内容,返回一个字符串
- `readline()`:每次只读一行,返回字符串
- `readlines()`:读取全部行,返回字符串列表

```python
# 示例:三种读法
# 先写入一个测试文件
f = open('poem.txt', 'w', encoding='utf-8')
f.write('床前明月光\n疑是地上霜\n举头望明月\n低头思故乡\n')
f.close()

# read():一次性读取全部内容
f = open('poem.txt', 'r', encoding='utf-8')
content = f.read()
print(content)
f.close()

# readline():每次只读一行
f = open('poem.txt', 'r', encoding='utf-8')
line1 = f.readline()
line2 = f.readline()
print(repr(line1))    # repr 显示原始字符串,能看到 \n
f.close()

# readlines():读取所有行,返回列表
f = open('poem.txt', 'r', encoding='utf-8')
lines = f.readlines()
print(lines)          # 每行是列表的一个元素
f.close()

# 逐行解剖
# ① read() 适合小文件,大文件一次性读取会占用大量内存
# ② readline() 内部维护一个读到哪了的指针
# ③ readlines() 返回的每个元素包含末尾的 \n
```

**常见错误**

> 1. **错误现象**:readlines 遍历时每行多一个空行
>    **原因:**每行末尾有 \n,print 又加一个 \n。修正:`line.strip()`
> 2. **错误现象**:大文件用 read() 导致内存溢出
>    **原因:**read() 一次性读入内存。修正:用 readline 或遍历

#### with 上下文管理器

> **痛点**

每次写文件都要记得 close(),
一旦忘记或中间报错退出,
文件就可能损坏或数据丢失。

> **类比**

进自动门:with 相当于走进去时自动开门,
走出来时自动关门,不管里面发生什么。

> **解释**

`with open(...) as f:` 创建上下文管理器,
退出代码块时自动关闭文件,即使发生异常也会关闭。

```python
# 示例:with 写法 vs 传统写法
# 传统写法:需要手动 close(),容易遗漏
f = open('old.txt', 'w', encoding='utf-8')
f.write('hello\n')
f.close()                    # 如果上面报错,这行永远不会执行

# with 写法:无论是否出错,退出时自动关闭
with open('new.txt', 'w', encoding='utf-8') as f:
    f.write('hello\n')
    f.write('world\n')
# 这里文件已经自动关闭,无需 f.close()

# 验证写入成功
with open('new.txt', 'r', encoding='utf-8') as f:
    print(f.read())

# 逐行解剖
# ① with 打开文件,将文件对象赋值给 f
# ② 在 with 代码块内操作 f
# ③ 离开 with 块时,Python 自动调用 f.close()
```

**常见错误**

> 1. **错误现象**:with 块外使用 f
>    **原因:**with 块结束后 f 已关闭。修正:在块内操作
> 2. **错误现象**:忘写冒号
>    **原因:**`with open(...) as f` → SyntaxError。修正:加冒号

> - with 块结束后,文件是打开还是关闭?
> - 如何在 with 中同时打开两个文件?

#### encoding=utf-8 的重要性

> **痛点**

打开别人写的含中文的文件,
或自己写的内容出现锟斤拷乱码,
往往就是编码没指定对。

> **类比**

编码就像翻译密码本:写的时候用密码本 A,
读的时候必须也用密码本 A,否则内容乱套。

> **解释**

Python 3 内部使用 Unicode 字符串,
但文件在硬盘上是字节流。
`encoding=utf-8` 告诉 Python
写入时用什么规则把字符变成字节,读取时再反向解码。

```python
# 示例:显式使用 utf-8 编码写入中文
with open('cn.txt', 'w', encoding='utf-8') as f:
    f.write('中文测试 ABC 123\n')

# 用 utf-8 读取
with open('cn.txt', 'r', encoding='utf-8') as f:
    print(f.read())

# 编码不一致会怎样
with open('mix.txt', 'w', encoding='utf-8') as f:
    f.write('汉字\n')

# 如果用错误的编码(如 ascii)读取,会抛出 UnicodeDecodeError
try:
    with open('mix.txt', 'r', encoding='ascii') as f:
        print(f.read())
except UnicodeDecodeError as e:
    print('编码错误:', e)

# 逐行解剖
# ① 不指定 encoding 时,Python 使用系统默认编码
# ② UTF-8 是目前互联网和 Linux/macOS 的通用编码
# ③ 读写双方必须用同一编码
```

**常见错误**

> 1. **错误现象**:Windows 上中文乱码
>    **原因:**Windows 默认 gbk,文件是 utf-8。修正:显式写 encoding='utf-8'
> 2. **错误现象**:读写编码不一致
>    **原因:**写用 utf-8,读用 gbk。修正:统一编码

> **问自己**:
> - 在 Windows 上如果省略 encoding 会怎样?
> - len(字符串) 返回的是字符数还是字节数?
> - 为什么建议所有文件都显式写 encoding=utf-8?

#### JSON 为何需要 + 类型对应

> **痛点**

列表/字典直接 write() 进文件会变成没有格式的字符串,
下次读回来不方便还原为结构化数据。

> **类比**

JSON 就像国际通用表格格式:
Python 写的 Excel,JavaScript 也能打开,
只要大家都按同样的规则打包。

> **解释**

JSON(JavaScript Object Notation)是一种轻量级数据交换格式,
本质是一个字符串。Python 的列表/字典可以和 JSON 字符串互相转换,
且多数语言都支持 JSON。

**类型对应表**

```
Python            JSON
dict       <->     object {}
list/tuple <->     array  []
str        <->     string ""
int/float  <->     number
True/False <->     true/false
None       <->     null
```

```python
# 示例:Python 类型与 JSON 类型的对应关系
import json

# Python 字典 -> JSON 对象 {}
student = {'name': '张三', 'age': 20, 'scores': [90, 85, 78]}
print('原始 Python 对象:', student)

# 用 json.dumps 把 Python 对象转为 JSON 字符串
json_str = json.dumps(student, ensure_ascii=False)
print('JSON 字符串:', json_str)
print('类型:', type(json_str))   # 注意:现在是 str 类型!

# 逐行解剖
# ① json.dumps(obj, ensure_ascii=False) 序列化为 JSON 字符串
# ② ensure_ascii=False 让中文不被转义成 \uXXXX
# ③ JSON 本质是字符串,传输和存盘都很方便
```

**常见错误**

> 1. **错误现象**:`json.dumps(set())` → TypeError
>    **原因:**set 不是 JSON 支持的类型。修正:转成 list
> 2. **错误现象**:中文变成 \uXXXX
>    **原因:**ensure_ascii 默认 True。修正:写 ensure_ascii=False

> **问自己**:
> - dumps 的 ensure_ascii 参数对中文有什么影响?
> - JSON 里的布尔值写法与 Python 有什么不同?
> - tuple 转 JSON 后再读回来是什么类型?

#### json.dump / json.load (文件读写)

> **痛点**

dumps/loads 拿到字符串后还要再手动写入文件,
多了一步。能不能直接从文件读写 JSON?

> **类比**

dumps 是先把行李箱打包好再叫快递;
dump 是直接让快递从你家门口把行李拉走,更省事。

> **解释**

`json.dump(obj, f)` 把 Python 对象序列化后直接写入文件;
`json.load(f)` 从文件读取并还原为 Python 对象。
比 dumps/loads 少写一层。

```python
# 示例:json.dump 和 json.load
import json

# 准备一个嵌套的 Python 字典
library = {
    'books': [
        {'title': 'Python入门', 'price': 59.0},
        {'title': '算法图解', 'price': 45.0}
    ],
    'total': 2
}

# json.dump:直接把对象写入文件
with open('library.json', 'w', encoding='utf-8') as f:
    json.dump(library, f, ensure_ascii=False, indent=2)
# indent=2 让 JSON 文件自动缩进,方便人类阅读

# json.load:从文件还原为 Python 对象
with open('library.json', 'r', encoding='utf-8') as f:
    data = json.load(f)

print('读回的对象:', data)
print('类型:', type(data))
print('第一本书:', data['books'][0]['title'])

# 逐行解剖
# ① json.dump(obj, f, ensure_ascii=False, indent=2) 写入文件
# ② indent 参数控制缩进,调试时写上更易读
# ③ json.load(f) 返回值类型由 JSON 内容决定
```

**常见错误**

> 1. **错误现象**:`json.dump(obj, 'path')` → TypeError
>    **原因:**第二个参数必须是文件对象,不是路径。修正:先 open
> 2. **错误现象**:文件内容不是合法 JSON,load 报错
>    **原因:**json.JSONDecodeError。修正:用 try-except 捕获

> **问自己**:
> - dump 和 dumps 有什么区别?
> - load 返回后,字典里的列表还是列表吗?
> - 如果文件不存在,load 会怎样?

#### json.dumps / json.loads (字符串转换)

> **痛点**

有时不存文件,而是要把数据结构通过网络发给别人,
或者从网络收到 JSON 字符串后解析,
这时就要用字符串转换版。

> **类比**

dump/load 是对文件柜操作(存取),
dumps/loads 是对纸条操作(收纸条、拆解纸条)。

> **解释**

`dumps` 把 Python 对象序列化为 JSON 字符串,
`loads` 把 JSON 字符串反序列化为 Python 对象。
比文件版多了一个 s,多操作一个字符串。

```python
# 示例:json.dumps 和 json.loads
import json

# dumps: Python 对象 -> JSON 字符串
fruits = ['苹果', '香蕉', '橘子']
s = json.dumps(fruits, ensure_ascii=False)
print('dumps 结果:', s)
print('类型:', type(s))

# loads: JSON 字符串 -> Python 对象
original = json.loads(s)
print('loads 结果:', original)
print('类型:', type(original))

# 典型应用场景:解析从网络收到的 JSON 字符串
response = '{"status": "ok", "count": 3, "items": ["a", "b", "c"]}'
result = json.loads(response)
print('状态:', result['status'])
print('数量:', result['count'])

# 逐行解剖
# ① json.dumps(obj) 返回一个字符串
# ② json.loads(s) 接收一个 JSON 字符串,还原为 Python 对象
# ③ 注意区分:dump/load 对文件,dumps/loads 对字符串
```

**常见错误**

> 1. **错误现象**:`json.loads("{'a': 1}")` → JSONDecodeError
>    **原因:**JSON 要求双引号,单引号不合法。修正:`'{"a": 1}'`
> 2. **错误现象**:混淆 dump 和 dumps
>    **原因:**多一个 s 就多一层字符串操作

> **问自己**:
> - dumps 和 dump 什么时候用哪个?
> - JSON 字符串里能用单引号吗?
> - loads 能解析空字符串吗?

#### 异常处理: try / except

> **痛点**

用户输入 'abc' 却让 int() 转换,
程序直接崩掉并抛出 ValueError,体验极差。

> **类比**

就像出门带把伞:如果不下雨(没异常)正常走,
如果下雨(异常)就撑伞(兜底处理),
不会淋湿(崩溃)。

> **解释**

try 块里放可能出错的代码,
except 块捕获异常并给出友好处理,
避免程序直接退出。

```python
# 示例:安全转换整数
def safe_int(value):
    """把输入转为整数,如果失败返回 None"""
    try:
        n = int(value)       # 如果 value='abc',这里抛出 ValueError
    except ValueError:
        print('输入的不是整数:', value)
        return None
    return n

# 执行过程
# safe_int('123'):int('123') 成功 → n=123 → 不进入 except → 返回 123
# safe_int('abc'):int('abc') 失败 → 匹配 ValueError → print → return None

print(safe_int('123'))    # 123
print(safe_int('abc'))    # 先打印错误信息,再输出 None

# 逐行解剖
# ① try: 后面的代码块是正常路径
# ② except 异常类型: 捕获特定异常,只处理已知的错误
# ③ 如果 try 里的代码都正常,except 不会执行
```

> **BREAK IT:except 不带类型**

```python
# ============ BREAK IT 演示 ============
# try:
#     1 / 0
# except:              # 捕获所有异常,包括 KeyboardInterrupt!
#     print("出错了")
# ============ END BREAK IT ============

# 原因:except 不带类型会捕获所有异常,容易隐藏真正的 bug
# 建议:始终写明异常类型,如 except ZeroDivisionError
```

**常见错误**

> 1. **错误现象**:`except:` 不带类型
>    **原因:**会捕获所有异常,容易隐藏 bug。修正:写明异常类型
> 2. **错误现象**:捕获了异常但没处理
>    **原因:**pass 吞掉异常,调试困难。修正:至少打印日志

> **问自己**:
> - 除零错误对应的异常类型是什么?
> - 一个 try 块后面可以跟多个 except 吗?
> - except 和 except Exception 有什么区别?

#### try / except / else / finally 完整语法

> **痛点**

只靠 try/except 不够:无论是否出错都要收尾工作
(如关闭文件),有时只有正常时才有后续操作。

> **类比**

吃火锅:try 是涮肉,except 是烫到了赶紧拿开,
else 是涮好了享受吃,finally 是吃完总得关火。

> **解释**

四个块的职责:
- try:放可能出错的核心代码
- except:出错时处理
- else:**没出错时才执行**,相当于奖励
- finally:**无论有无异常都执行**,常用于清理

```python
# 示例:完整语法演示
def parse_number(text):
    print('--- 开始处理:', repr(text), '---')
    try:
        result = int(text)              # 可能出错
    except ValueError:
        print('except: 不是数字')        # 只有出错时执行
    else:
        print('else: 转换成功,结果为', result)  # 只有成功时执行
    finally:
        print('finally: 收尾工作,每次都执行')   # 永远执行
    print()

# 调用 '42' 时:
#   try 中 int('42') 成功 → 跳过 except → 执行 else → 执行 finally
# 调用 'abc' 时:
#   try 中 int('abc') 失败 → 执行 except → 跳过 else → 执行 finally

parse_number('42')
parse_number('abc')

# 逐行解剖
# ① else:当 try 块没有异常时执行
# ② finally:即使 try/except 里有 return,finally 也会在返回前执行
```

**常见错误**

> 1. **错误现象**:把收尾代码放在 try 最后
>    **原因:**如果 try 中异常,收尾代码不会执行。修正:放 finally
> 2. **错误现象**:finally 里写 return
>    **原因:**会覆盖 try/except 里的 return。修正:避免

> **问自己**:
> - else 和把代码放在 try 最后有什么区别?
> - finally 什么时候执行?
> - finally 里写 return 会怎样?

#### 常见异常类型速查

> **痛点**

程序报错时看不懂错误信息,不知道从何查起。

> **类比**

异常类型就像病历上的病名:
知道病名才能对症下药,
每一种异常对应一类问题。

> **解释**

Python 内置几十种异常,
初学者先掌握最常见的 6 种。

**异常类型与错误对照表**

- **ValueError** - 类型对,值不对,如 int('abc')
- **TypeError** - 类型不对,如字符串加整数
- **KeyError** - 字典中没有这个键
- **IndexError** - 列表/字符串索引越界
- **FileNotFoundError** - 文件路径不存在
- **ZeroDivisionError** - 除数为零

```python
# 示例:常见异常类型演示
# ValueError: 类型正确但值不合法
try:
    int('hello')
except ValueError as e:
    print('ValueError:', e)

# TypeError: 类型不对
try:
    '2' + 2
except TypeError as e:
    print('TypeError:', e)

# KeyError: 字典键不存在
try:
    d = {'a': 1}
    d['b']
except KeyError as e:
    print('KeyError:', e)

# IndexError: 索引超出范围
try:
    lst = [1, 2, 3]
    lst[10]
except IndexError as e:
    print('IndexError:', e)

# FileNotFoundError: 文件不存在
try:
    open('ghost.txt', 'r')
except FileNotFoundError as e:
    print('FileNotFoundError:', e)

# ZeroDivisionError: 除零
try:
    1 / 0
except ZeroDivisionError as e:
    print('ZeroDivisionError:', e)
```

#### 综合练习 —— 异常安全加载 JSON 文件

> **痛点**

实际项目中经常要读取配置文件/数据文件,
文件可能不存在、可能是空文件、可能内容不是合法 JSON,
任何一种情况都会导致程序崩溃。

> **类比**

就像开门取快递:门可能锁了(文件不存在),
快递可能丢了(空文件),快递可能被压坏(内容损坏),
每种情况都要预案。

> **解释**

把文件读取 + JSON 解析 + 异常处理组合在一起,
一次性应对多种错误,是实际开发最常用模式。

```python
# 示例:异常安全加载 JSON 文件
import json

def safe_load_json(path, default=None):
    """安全加载 JSON 文件,任何异常都返回 default"""
    try:
        with open(path, 'r', encoding='utf-8') as f:
            return json.load(f)
    except FileNotFoundError:
        print(f'[警告] 文件不存在: {path}')
    except json.JSONDecodeError:
        print(f'[警告] JSON 格式错误: {path}')
    except Exception as e:
        print(f'[警告] 未知错误: {e}')
    return default

# 测试 1: 文件不存在
result = safe_load_json('不存在的文件.json', default={})
print('测试1 结果:', result)

# 测试 2: 合法 JSON
with open('ok.json', 'w', encoding='utf-8') as f:
    f.write('{"name": "小明", "score": 90}')
result = safe_load_json('ok.json', default={})
print('测试2 结果:', result)

# 测试 3: 内容损坏
with open('bad.json', 'w', encoding='utf-8') as f:
    f.write('这不是 JSON')
result = safe_load_json('bad.json', default={})
print('测试3 结果:', result)

# 逐行解剖
# ① safe_load_json 是实际项目的常见封装
# ② json.JSONDecodeError 是 ValueError 的子类
# ③ 多个 except 的顺序:从具体到宽泛
# ④ with + json.load(f) 组合,保证文件正确关闭
```
