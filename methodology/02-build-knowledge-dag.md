# 02 · 知识地图推理 —— 如何从零推导出终点能力所需的全部知识点

> **归属阶段:** PLAN (状态机 `PLAN` 态核心参考)
> **用途:** 给定一个"终点能力",系统化地倒推出完整的知识链条,构建有向无环图(DAG),做拓扑排序得到学习顺序。
> **适用场景:** 任意知识域的课程设计、知识点审计、学员诊断(哪块基础弱)。

---

## 1. 推理三步法(倒推法)

```
终点能力 → 第一层前置 → 第二层前置 → ... → 原子知识点(已掌握的起点)
```

### Step 1 · 写出终点能力的直接前置

问自己:"要掌握 X,必须已经会什么?"列出所有直接前置。

示例 —— 终点能力"微调 LLM":
- 要微调 LLM → 需要理解 **LoRA / QLoRA** 的原理
- 要微调 LLM → 需要掌握 **PyTorch 训练循环**
- 要微调 LLM → 需要理解 **Tokenizer + Attention Mask**
- 要微调 LLM → 需要能从 HuggingFace 加载预训练模型

### Step 2 · 递归展开每个前置

对每个前置,重复问"要掌握它,必须已经会什么",直到触及学员已掌握的原子知识。

以"LoRA 原理"为例:
- LoRA → 理解矩阵低秩分解 → 理解矩阵乘法 → 理解 **NumPy / 张量运算**
- LoRA → 理解微调 vs 全量微调 → 理解 **梯度下降 / 反向传播**
- LoRA → 理解 HuggingFace PEFT 库 → 理解 **Python 类与对象**

### Step 3 · 合并、去重、拓扑排序

把所有"展开后的节点 + 边"合并成有向无环图(DAG),做拓扑排序 → 得到学习顺序。

### 为什么倒推而不是顺推?

顺推(从 print 开始一点点讲)的问题:
- **不知道为什么要学**: 学到函数时不知道这跟目标有什么关系
- **动力衰减**: 60 天时前 20 天还没见到目标,流失率高
- **冗余**: 把"永远用不到的知识"也塞进来

倒推的优势:
- **目标清晰**: 每个知识点都能讲清楚"学这个是为了..."
- **最小必要**: 只教终点能力真正需要的,不贪全
- **支撑课程压缩**: 21 天→10 天压缩的关键就是"倒推后只保留必要链"

## 2. 知识依赖图的构建方法

### 节点定义

每个节点是一个"知识点",粒度参考:
- ✅ "列表 CRUD" (具体到操作层面)
- ✅ "梯度下降原理" (具体到理解层面)
- ❌ "Python 基础" (太粗,不可执行)
- ❌ "会写 for 循环" (太细,可合并到"控制流")

### 边定义

A → B 表示"必须先掌握 A,才能学 B"。
三种依赖强度:
- **强依赖(strong)**: 不会 A 完全无法学 B (例: 不会矩阵乘法 → 无法学 Attention)
- **弱依赖(weak)**: 不会 A 也能学 B,但理解不深 (例: 不会 Vim → 也能学 Python)
- **可同步(parallel)**: 两节课放在同一天的不同时段,互相不依赖

### 拓扑排序算法

```
1. 找到所有"入度为 0"的节点(没有前置) → 作为第 1 天
2. 移除这些节点,重新计算入度
3. 重复,直到所有节点被排序
4. 若剩余节点入度都 > 0 → 说明有环,需要检查依赖定义
```

**实操建议**: 用 web search 验证关键依赖关系的合理性(藤校课程大纲、fast.ai 等成熟课程),不要纯靠 LLM 拍脑袋。详见 `03-design-learning-sequence.md` 编排原则。

---

## 3. 每个知识点的"最低要求"标注

不是所有知识点都需要"深入掌握"。对每个节点标注:

| 标注 | 含义 | 习题深度 |
|---|---|---|
| **了解** | 能说出是什么、解决什么问题 | ⭐⭐ 题(单步变形) |
| **能写** | 能独立写出代码实现 | ⭐⭐⭐ 题(组合应用) |
| **能教** | 能给别人讲清楚原理 | ⭐⭐⭐⭐ 题(抽象建模) |

**标注依据:** `requirements.md` 里的 `depth_goal` 字段。`depth_goal = 能写` → 整张图的默认最低要求是"能写",只有真正需要深入的节点升级到"能教"。

---

## 4. DAG 剪枝规则("最小必要链")

优化目标:终点能力真正需要的知识点数最少。

**剪枝规则:**
1. 若某知识点从任一终点能力节点出发不可达 → 裁剪
2. 若某知识点被更具体的知识点完全覆盖(父节点被子里节点覆盖) → 裁剪父节点
3. 弱依赖的叶节点,如果标注为"了解"且时间预算紧 → 可裁剪,放到"可选资料"

**每个被剪掉的节点必须记录到 `learning-plan.md` 的 `pruned_branches` 字段**,说明裁剪理由。用户可以人工补回。

---

## 5. DAG 正确性验证(关键,因为这是整个产品的 #1 可行性风险)

知识依赖常常是 tacit、有争议、非线性的。LLM 会幻觉出"看起来合理但实际错误"的前置链。

### 验证方法

| 步骤 | 操作 |
|------|------|
| 1 | 用 web search 找到 1-2 门目标域的成熟课程(Coursera / fast.ai / 官方文档 tutorial) |
| 2 | 提取其章节顺序作为 proxy gold-set |
| 3 | 把我们推理出的 DAG 和 gold-set 对比 |

### 阈值(来自设计文档 Success Criterion #3)

- **零 critical-path 错误**: critical-path 依赖 = 移除后导致 target 节点从 start 节点不可达的 strong 边
- **≤10% 非关键依赖遗漏**

### 如果验证失败

回 PLAN 态重新推理,标注分歧点。不要带着已知错误的 DAG 进入 BUILD 态。

## 6. 实操示例结构

输出 `learning-plan.md` 时,DAG 部分应包含:

```markdown
## 知识依赖图(DAG)

### 节点(共 N 个)
| id | 知识点 | 粒度 | 最低要求 |
|----|--------|------|----------|
| n1 | Python 基础语法 | 操作级 | 了解 |
| n2 | 函数与模块化 | 操作级 | 能写 |
| ... | ... | ... | ... |

### 边(共 M 条)
| from | to | 强度 | 备注 |
|-------|-----|------|------|
| n1 | n2 | strong | 不会语法无法写函数 |
| n2 | n5 | strong | ... |

### 拓扑排序结果
第 1 天: [n1]
第 2 天: [n2, n3]  (parallel,互不依赖)
第 3 天: [n5]
...

### 被裁剪的枝
- nX "装饰器": 从终点能力不可达,裁剪
- nY "元类": 弱依赖叶节点,时间预算紧,裁剪 → 放入可选资料
```

## 6. 教学法标注与编排原则落地(v1.1 新增)

> 来自 R2 迭代:PLAN 产出必须显式标注编排原则和教学法,不能仅靠 agent 内部判断。

### 6.1 每个 DAG 节点必须标注 `teaching_method`

| 值 | 适用场景 | 触发 |
|---|---|---|
| `ncdl` | 有常见反模式的节点(const 不可变误解、var hoisting 异常) | 节点有"易错点"特性 |
| `dual_layer` | 通过叙事锚点承载的节点(电商系统里的 OOP) | 节点需要业务案例 |
| `consumer_gate` | 设计意图需要被消费者函数强约束的节点 | 节点是验收核心 |

未标注 = 使用默认教学法(纯讲解)。

### 6.2 lesson_plan 必须包含 `pedagogy_notes`

每条 lesson 记录体现了哪个编排原则:

| 原则 | 在 pedagogy_notes 怎么写 |
|------|--------------------------|
| 工具先行 | "先教 X 立刻能做 Y,理论后置" |
| 脚手架递进 | "每节只引入一个新抽象层" |
| 螺旋复访 | "核心概念 X 在本课复习一次,后续模块还会再出现" |

**为什么:** PLAN 产出的 DAG 应该在产物中就体现编排意图,而不是依赖 BUILD 自己重新判断。

### 6.3 剪枝必须 user-validated

agent 不能自行决定剪枝。剪枝后必须在 AWAIT_CONFIRM 阶段让用户看到:

> 剪掉了 3 个节点: TDZ 深入、数据类型、解构赋值。要不要补回?(auto-skip / 补回 X / 全部补回)

只有用户 approve 后才能进入 BUILD。
