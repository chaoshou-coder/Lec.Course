# 06 · 学科特定教法模式库 —— 各知识域的反模式与教学设计范式

> **归属阶段:** BUILD + QA (build subagent 制课参考 + QA 内容正确性检查依据)
> **用途:** 这套文档从 OOP 教学设计提炼出通用教学法,然后把各学科域(ML/DL/爬虫/AI 应用等)的"反模式 + 教学设计范式"汇聚成模式库。**OOP 是 pattern #1,其他学科续之。**
> **适用场景:** build subagent 为某知识点域制课时查找对应的教学陷阱和 NCDL 设计模板。

## 通用教学法(适用于所有学科域)

以下三种教学法从 OOP 设计提炼,推广到全项目。

### 1. 双层覆盖教学法(Dual-Layer Coverage)

**叙事锚点(80% 教学时间):** 用一个可演进的业务系统贯穿整个模块,语法点被"需求升级"逼出来。

**语法点独立样本(20% 教学时间):** 无法嵌入叙事的语法点用最小可复现实例单独演示(1-2 行代码 + 类比)。

**适用:** 所有"语法体系 vs. 真实应用"有落差的模块。

### 2. NCDL — 负案例驱动学习(Negative-Case-Driven Learning)

四步循环:
1. **Code-Along(正向):** 演示"正确的做法"
2. **Break It(故意破坏):** 演示"这样写能跑通但有问题"
3. **读 Traceback / 出问题:** 学员读报错、测出 bug、看效果变差
4. **Fix & Discussion:** 讨论"为什么错了" → 引出防御性编程知识

**与普通"错题本"的区别:** 错题本是学生做错后复盘;NCDL 是教师**在学生犯错之前先展示"常见错法"**,学员有了免疫力。

**不适用:** 纯数学/概念(反向传播推导没有"错法"给学生 break)。

### 3. 消费者函数即门控(Consumer-Function Gate)

**不检查学生代码里有没有 if-elif;让消费者函数要求学生代码必须是多态的。**

给一个**结构性约束强的骨架**(消费者函数 ≤4 行),学员只能通过正确实现来完成。

```python
# === 教师提供,不可修改 ===
def checkout(cart_total, payment):
    """
    约束:本函数不使用 if/elif/isinstance/type()
    约束:本函数体不超过 4 行
    """
    # TODO:补全这 4 行(学员写)
    ...

# === 学员代码区 ===
# 实现 Alipay / WeChatPay / ApplePay 子类
```

**门控本质:** 用**骨架代码的结构约束**取代**事后反模式检查**。

## Pattern 1: OOP(面向对象编程)

### L1-L4 认知进阶模型(替代传统"三支柱")

传统三支柱(封装/继承/多态)不是层层递进 —— 多态和继承是平级概念,谁先谁后都能讲,学员没有认知阶梯。

```
L1 数据与行为打包(⭐⭐)
   解决:数据与方法分离 → 如何把相关的东西合在一起
   认知跃动:从"过程式"到"对象化"

L2 代码复用(⭐⭐⭐)
   解决:重复代码 → 如何通过继承减少复制
   认知跃动:从"每个对象独立描述"到"抽取共性"

L3 行为契约(⭐⭐⭐⭐)
   解决:同一接口不同行为 → 如何写一个处理多类型的函数
   认知跃动:从"知道每个类怎么用"到"设计让代码可扩展"
   ⚠️ L2→L3 跃变最难:视角从"类自身"切换到"消费者函数"

L4 对象交互协议(⭐⭐⭐⭐⭐)
   解决:如何让对象"像内建类型一样工作"
   认知跃动:从"用 OOP 实现功能"到"用 OOP 写出 Pythonic 代码"
```

### 业务演进叙事:电商订单系统四天

```
Day 05(L1) "订单系统太乱了"      → 散落的 dict → Product + Order 类
Day 06(L2) "要上电子书改不过来"  → 复制粘贴 → Product 基类 + 子类
Day 07(L3) "要接支付宝和微信"    → if-elif 又臭又长 → Payment(ABC) + 多态
Day 08(L4) "两个购物车能合并吗"   → 手写 for 循环 → cart1 + cart2 一行搞定
```

### 语法点对照表:哪些能进叙事锚点,哪些需要独立场景

| L | 语法点 | 能嵌入叙事? | 处理方案 |
|---|---|---|---|
| L1 | class / __init__ / self | ✅ Product 类 | 主线 |
| L1 | @property / __str__ | ✅ Product.price 不允许负 / 打印订单 | 主线 |
| L2 | class Child(Parent) / super() | ✅ DigitalProduct / PhysicalProduct | 主线 |
| L2 | MRO 查找顺序 | ⚠️ 无电商对应 | 独立最小示例 |
| L3 | 鸭子类型多态 / abc.ABC | ✅ Payment 契约 | 主线 |
| L3 | isinstance 反模式 | ⚠️ 作为"为什么不用"引入 | 独立最小示例 |
| L4 | __add__ / __len__ / __iter__ | ✅ Cart 合并 / len(cart) | 主线 |
| L4 | 描述符 / __slots__ / 元类 | ❌ 无电商对应 | 仅资料提及,不考 |

### L2→L3 突破方法(痛点重构,10+10+5+20 分钟)

```
Step 1(痛点):给学员 120 行 if-elif 支付代码,要求新增 Apple Pay
  → 学员发现要在 N 个 elif 外加新分支,改核心函数 → 引出"开闭原则"需求
Step 2(重构):引导学员"如果每种支付方式自己知道怎么 pay"
  → 核心函数只需要:payment.execute(amount) 一行
Step 3(契约):新增支付方式但不提供 pay() 方法
  → 学员发现运行时才报错 → 引出 abc:用契约锁定
Step 4(破坏式实操):学员在给定的消费者函数骨架下完成题目
```

### NCDL 示例(OOP 反模式)

- Break It = 故意漏写 self → `TypeError: missing 1 required positional argument`
- Break It = 故意不调用 super() → 子类丢了父类属性
- Break It = 故意不实现抽象方法 → `TypeError: Can't instantiate abstract class`
- Break It = 用 isinstance 做类型判断 → 运行时多态失效

## 其他学科域的模式(待 build subagent 填充)

当 build subagent 为 ML / DL / Web Scraping / AI 应用等模块制课时,按以下模板追加反模式和 NCDL 示例:

### ML 反模式
- 故意不做数据标准化,看模型精度崩
- 故意把测试集混入训练集,看精度虚高
- 故意漏 `train_test_split`,在全量数据上评估

### DL 反模式
- 故意忘 `optimizer.zero_grad()`,看梯度累积
- 故意不 `model.eval()`,看验证集方差
- 故意在推理时没 `torch.no_grad()`,看显存爆炸

### Web Scraping 反模式
- 故意不加 User-Agent,看 429
- 故意爬过快,看 IP 被封
- 故意不处理 encoding,看中文乱码

### AI 应用 反模式
- 故意把 API Key 硬编码,看 GitHub 推送后的安全警告
- 故意不做 rate limiting,看账号被封
- 故意在 production 用 print 调试,看日志爆炸

## QA 深度检查要求(v1.2 R10)

> QA 不只是"查文件是否存在",而是四维深度检查。

### 1. 知识性内容正确性

QA 随机抽 2-3 个知识点,验证其定义/示例/解释是否准确:

| 检查项 | 方法 |
|--------|------|
| 定义准确性 | 对照 MDN/官方文档,确认 notes.md 中的定义无误 |
| 示例可运行 | 复制 code 示例到控制台/Node.js,确认输出与注释一致 |
| 类比恰当性 | 确认类比不会导致误解(如"const = 密封盒"需注明"内容可变") |

### 2. 8 步循环验证

structural_check 必须验证 notes.md 包含 8 个关键段:

| # | 关键段 | 检查方式 |
|---|--------|---------|
| ① | 痛点(为什么需要) | 搜索 `> **痛点**` |
| ② | 类比(生活例子) | 搜索 `> **类比**` |
| ③ | 解释(是什么) | 搜索 `> **解释**` |
| ④ | ASCII 内存图 | 搜索 `ASCII 内存图` 或 ` ``` ` |
| ⑤ | 执行过程跟踪 | 搜索 `执行过程` |
| ⑥ | 常见错误 | 搜索 `> **常见错误**` |
| ⑦ | 学员代码区 | 搜索 `学员代码区` |
| ⑧ | 参考答案 | 搜索 `参考答案` |

### 3. pedagogy_notes 对照

QA 检查 PLAN 的 `pedagogy_notes` 是否在 BUILD 中落地:

| pedagogy_notes 声明 | BUILD 中应看到 |
|---------------------|---------------|
| "工具先行" | Day 1 第一个代码示例就能运行 |
| "脚手架递进" | 每节只引入一个新抽象层(不跳跃) |
| "NCDL" | Break It 演示段 + 反模式练习 |
| "消费者门控" | ≤4 行消费者函数 + 多态调用 |

### 4. 学员视角评估

QA 从学员角度评估:

| 评估项 | 方法 |
|--------|------|
| 练习可达性 | 学员能在预计时间内完成吗?(★=5min,★★=10min,★★★=15min) |
| 难度线性 | 6 道练习难度是否 ★→★★→★★★ 递增? |
| 类比可理解 | 类比是否来自学员已知经验(不引入未知概念)? |
| 先修知识 | 需要的先修知识是否都在前面课时覆盖? |
